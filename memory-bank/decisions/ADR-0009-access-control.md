# Система управления доступом

## Цель

Планируется ввести единую систему управления доступом, которая:

- сохраняет совместимость с текущим `is_admin`
- задает доступ через стабильные `access_scope_id`
- централизует проверку прав на уровне backend routing
- работает по принципу `default deny`
- допускает дальнейшее расширение до объектных ограничений без переделки основы

На первом этапе объектные ограничения не рассматриваются. Доступ определяется только глобальными ролями пользователя и матрицей прав по областям системы.

## Базовая модель

Система состоит из трех основных элементов.

### `is_admin`

`is_admin` сохраняется и трактуется как системный override полного доступа.

Назначение:

- аварийный и системный доступ
- обратная совместимость с текущей auth-моделью
- возможность не блокировать администрирование на этапе миграции

`is_admin` не является обычной ролью из матрицы прав. Это отдельный bypass, который дает полный доступ ко всем scope.

### `primary_role`

У каждого пользователя есть одна основная роль `primary_role`.

Особенности:

- primary roles описаны в коде
- во время выполнения они доступны только для чтения
- они существуют всегда и не зависят от содержимого БД
- их можно сразу назначать пользователям

`primary_role` задает базовый набор прав пользователя.

### Дополнительные роли

Помимо `primary_role` пользователю могут назначаться дополнительные роли из БД.

Назначение дополнительных ролей:

- расширять доступ относительно `primary_role`
- собирать прикладные комбинации прав
- подготавливать основу для следующего этапа, где роли смогут назначаться и на конкретные объекты

Итоговый доступ пользователя рассчитывается так:

1. Если `is_admin = true`, пользователь получает полный доступ.
2. Иначе берутся права `primary_role`.
3. Добавляются права всех дополнительных ролей пользователя.
4. Если право отсутствует, доступ запрещен.

## Основная единица авторизации

Ключевая единица доступа — `access_scope_id`.

`access_scope_id` — это стабильный идентификатор области системы, соответствующий бизнес-сущности, модулю, use case, projection или системному разделу.

Важно:

- `access_scope_id` не должен зависеть от URL
- `access_scope_id` должен использовать уже существующую систему именования проекта
- новая система идентификаторов не вводится без необходимости

Поэтому формат рекомендуется брать из существующих project ids, например:

- `a024_bi_indicator`
- `a025_bi_dashboard`
- `a018_llm_chat`
- `u504_import_from_wildberries`
- `p900_mp_sales_register`
- `system_users`
- `system_tasks`

Главный принцип:

`access_scope_id` должен быть одинаковым в metadata, backend policy, БД и админке.

## Режимы доступа

На первом этапе используется простой набор режимов:

- `none`
- `read`
- `all`

Смысл режимов:

- `none` — доступа нет
- `read` — только чтение
- `all` — чтение и изменение в рамках данного scope

`all` на первом этапе сознательно остается укрупненным. В будущем его можно разложить на более точные действия:

- `write`
- `execute`
- `manage`
- `delete`
- `publish`

Но в первой версии отдельные capabilities не обязательны.

## Матрица прав

Основой системы является матрица:

`role x access_scope_id -> access_mode`

Если записи нет, это эквивалентно `none`.

Примеры:

- `viewer -> a025_bi_dashboard = read`
- `manager -> a025_bi_dashboard = all`
- `operator -> u504_import_from_wildberries = all`
- отсутствие записи для `viewer -> u504_import_from_wildberries` означает запрет

## Связь с endpoint

Endpoint не хранит право внутри себя и не является основной сущностью авторизации.

Каждый route привязывается к policy:

- `access_scope_id`
- `required_mode`
- `is_public`

Примеры:

- `GET /api/a025-bi-dashboard/:id -> a025_bi_dashboard, read`
- `POST /api/a025-bi-dashboard/upsert -> a025_bi_dashboard, all`
- `POST /api/u504/import/start -> u504_import_from_wildberries, all`

Таким образом backend проверяет не URL как таковой, а policy маршрута.

## Почему не права на URL

Привязка прав напрямую к URL слишком хрупкая:

- переименование маршрута не должно ломать модель доступа
- один scope может обслуживаться несколькими endpoint
- один и тот же business scope может иметь list/get/upsert/delete маршруты

Поэтому:

- `route` — это технический транспортный слой
- `access_scope_id` — это стабильная бизнес-область контроля доступа

## `capability`

На первом этапе отдельная сущность `capability` не требуется.

Сейчас достаточно пары:

- `access_scope_id`
- `access_mode`

Если позже появится необходимость более тонкого разделения, `capability` станет детализацией внутри scope. Тогда модель эволюционирует в:

`role x access_scope_id x capability`

Но первая версия остается проще:

`role x access_scope_id -> access_mode`

## Интеграция в metadata

`access_scope_id` должен быть встроен в `crates/contracts/src/**/**/metadata.json` как часть метаданных сущности.

Рекомендуемая структура:

```json
"access": {
  "scope_id": "a025_bi_dashboard",
  "modes": ["read", "all"],
  "operations": [
    { "id": "list", "required_mode": "read" },
    { "id": "get", "required_mode": "read" },
    { "id": "upsert", "required_mode": "all" },
    { "id": "delete", "required_mode": "all" }
  ]
}
```

Назначение этого блока:

- зафиксировать принадлежность сущности к `access_scope_id`
- описать допустимые режимы доступа
- задать логические операции, которые затем маппятся на route handlers

`metadata.json` не должен содержать конкретные URL. Он описывает бизнес-уровень доступа, а не транспортный слой.

## Primary roles

В систему вводится небольшой встроенный каталог основных ролей.

Они:

- описаны в коде
- имеют стабильные коды
- доступны приложению без чтения из БД
- не редактируются через runtime UI

Примерный набор:

- `admin`
- `manager`
- `operator`
- `viewer`

Это только базовые предустановленные роли. Они не отменяют роли из БД, а используются как обязательная основа для назначения пользователям.

## Роли из БД

Роли из БД являются расширениями встроенной модели.

Они нужны для:

- создания проектных или временных ролей
- прикладных комбинаций прав
- адаптации системы без перекомпиляции

Пользователь всегда имеет:

- один `primary_role` из кода
- ноль или более дополнительных ролей из БД

## Минимальная структура данных

Минимальный состав таблиц:

### `sys_roles`

- `id`
- `code`
- `name`
- `is_system`

`is_system` позволяет отличать встроенные роли от прикладных ролей БД, если потребуется зеркалирование встроенного каталога в БД.

### `sys_users`

- `id`
- `primary_role_code`
- `is_admin`

### `sys_user_roles`

- `user_id`
- `role_id`

### `sys_role_scope_access`

- `role_id`
- `access_scope_id`
- `access_mode`

На следующем этапе могут быть добавлены объектные назначения:

- `sys_user_role_object_access`
или
- `sys_object_acl`

Но это не входит в первую фазу.

## Принципы безопасности

Система строится по следующим правилам:

### `default deny`

Все endpoint считаются закрытыми, если явно не помечены как публичные.

### Единый реестр route policy

Каждый route должен регистрироваться только вместе с policy.

Нельзя добавлять новый route без указания:

- `access_scope_id`
- `required_mode`
- `is_public`

### Backend как единственный источник истины

Frontend может скрывать элементы UI, но решение о доступе принимает только backend.

### Отсутствие записи означает запрет

Если для роли нет записи в матрице прав по данному scope, доступ запрещен.

## Контроль целостности системы

Целостность системы должна обеспечиваться не ручной дисциплиной, а проверяемой архитектурой.

Обязательные механизмы:

- все route регистрируются только через обертку, требующую policy
- все используемые в route policy `access_scope_id` должны существовать в metadata
- все непубличные endpoint должны требовать авторизацию
- список публичных endpoint должен быть явным whitelist
- автоматические тесты должны проверять:
  - что нет route без policy
  - что нет неизвестных `access_scope_id`
  - что без токена защищенный route возвращает `401`
  - что без нужного права route возвращает `403`

Главный принцип контроля целостности:

в системе не должно быть технической возможности зарегистрировать новый endpoint без описания его access policy.

## Порядок внедрения

1. Сохранить `is_admin` как системный override.
2. Ввести встроенный каталог `primary_role`.
3. Добавить дополнительные роли из БД.
4. Ввести матрицу `role x access_scope_id -> access_mode`.
5. Добавить блок `access` в `metadata.json`.
6. Построить единый реестр route policy.
7. Перевести backend на `default deny`.
8. Добавить тесты полноты покрытия policy.
9. После стабилизации перейти ко второму этапу: объектные ограничения.

## Итог

Целевая система доступа выглядит так:

- `is_admin` сохраняется как полный системный override
- каждый пользователь имеет один встроенный `primary_role`
- пользователь может иметь дополнительные роли из БД
- роли получают права не на URL, а на `access_scope_id`
- `access_scope_id` использует уже существующую систему именования проекта
- права задаются матрицей `role x scope -> mode`
- каждый route привязан к `scope + required_mode`
- backend работает по `default deny`
- `metadata.json` становится источником описания бизнес-областей доступа
- объектные ограничения будут добавлены позже поверх этой модели, не ломая основу
