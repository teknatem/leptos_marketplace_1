# ADR-0010: Роли источников данных (прозрачный семантический слой)

Статус: принято; LLM-first Фаза 1 реализована 2026-06-27.
Дата: 2026-06-26.
Контекст-план: `Фаза 0` (см. план модернизации семантического слоя).

## Проблема

В приложении сосуществуют **три независимых способа** получить аналитические данные, и это сейчас
**непрозрачно** — у каждого свой реестр, своя поверхность, нет единых правил «чем пользоваться»:

- `dsNN` — декларативные схемы (`DataSourceSchema`) + универсальный `QueryBuilder`;
- `DataView dv001–dv007` — Rust-модули с собственным SQL, два периода, кэш;
- сырой `execute_query` — произвольный SELECT у LLM.

Симптомы непрозрачности: LLM видит только `list_data_views` (обнаружение, без чтения строк); плагины
ходят сырым `host.db.query`; одну и ту же `p904_sales_data` читают и `dv001`, и `ds03`, определяя
метрики по-разному; новые потребители (презентации) источника данных пока не имеют.

## Решение

Оставляем **три движка независимыми** (без конвергенции, без переписывания DataView в декларатив) и
вводим **прозрачную модель трёх ролей** + **единую витрину доступа** над ними. Каждый источник явно
помечен своей ролью; потребитель (человек или LLM) выбирает роль по понятным правилам.

### Три роли (две оси: *общность* и *готовность*)

| Роль | Что это | Ось | Когда | Движок | Потребители |
|---|---|---|---|---|---|
| **Base schema `dsNN`** | Декларативное описание одной канонической таблицы/проекции (поля, типы, joins, `can_group/aggregate/filter`) | широкий / «собери сам» | Гибкий ad-hoc: произвольные группировки/фильтры/агрегаты по «сырым» полям | universal_dashboard `QueryBuilder` | pivot-UI, дата-блоки плагинов/презентаций, «стандартные» запросы LLM, исследование |
| **DataView `dvNN`** | Курируемая «виртуальная таблица»: составные метрики, дефолтные измерения, **2 периода**, кэш | узкий / «готово к работе» | Благословлённые показатели и 2-периодные отчёты | Rust-функции `dv00x` | BI-индикаторы a024/a025, drilldown, 2-периодные отчёты, дашборды |
| **Raw SQL** | Произвольный SELECT | неограниченный / «вручную» | Нестандартные/разовые случаи, длинный хвост | `execute_query` (укреплённый) | LLM long-tail, power-users |

### Дерево выбора (нормативное)

1. Нужен **благословлённый показатель**, **сравнение 2 периодов** или **составная метрика**
   (`revenue = customer_in + customer_out`, GL turnover `CASE`) → **`dvNN`**.
2. Иначе нужен **ad-hoc разрез** по «сырым» полям одной канонической таблицы (группировка/фильтр/
   агрегат) → **`dsNN`**.
3. Не покрывается двумя выше (нестандартные join'ы, разовый аудит) → **Raw SQL** (с governance).

Калибровка на реальных запросах: «выручка по дням, 2 периода» → `dv`; «продажи по артикулу за месяц»
→ `ds`; «разовая сверка остатков по странному условию» → `SQL`.

### Правило перекрытия (дублирование — осознанное, не случайное)

Перекрытие источников по одной таблице **допустимо только если роли реально разные**. Эталон
by-design: `ds03` = гибкий доступ к `p904`; `dv001` = курируемая 2-периодная «выручка» по `p904`.
Запрещено плодить **одинаковые** определения метрик в `ds` и `dv` «просто так». Случайные/legacy
дубли (см. инвентаризацию) выводятся.

## Единая витрина доступа (прозрачность поверх трёх движков)

Движки внутри остаются раздельными, но наружу — **один вход**:

1. **Единый каталог источников** — один список всех трёх видов:
   ```
   DataSourceCatalogItem {
     kind: "base" | "dataview" | "raw",
     id, name, description,
     capabilities: { dimensions[], metrics[], filters[], supports_2period: bool, ad_hoc: bool }
   }
   ```
   Любой (и LLM) видит всё меню сразу и понимает роль каждого источника.

2. **Тонкий типизированный фасад** — единая точка входа и governance, но без универсального DSL:
   `query_schema`, `run_data_view_scalar`, `run_data_view_drilldown`, `execute_raw_query`. Каждый метод
   сохраняет честный контракт своего движка; табличные запросы возвращают общий `TabularResult`, а
   DataView продолжает возвращать существующие `IndicatorValue`/`DrilldownResponse`. Это маршрутизация
   и общая политика доступа, а НЕ слияние движков и НЕ переписыватель запросов.

Эффект: LLM/плагины/презентации получают **один каталог и одно дерево выбора** — обнаружить источник →
выбрать роль → вызвать соответствующий типизированный метод; роль каждого источника явная.

## Инвентаризация источников (сверено с кодом)

| Таблица/проекция | `dsNN` | `dvNN` | Прочее | Потребители | Статус |
|---|---|---|---|---|---|
| `p904_sales_data` | `ds03` (сырые поля + агрегаты) | `dv001` (revenue=customer_in+customer_out, cost, order_count, avg_check, commission, profit; 2 периода, кэш) | — | a024/a025 (через dv001), pivot (ds03) | **перекрытие by design** |
| `p903_wb_finance_report` | `ds01` (поля WB-финотчёта) | — | `d400` (хардкод monthly summary, p903+p914) | ds01-execute; d400-handler | `d400` — **legacy на вывод** |
| `p900_sales_register` | `ds02` | — | — | ds02-execute | ok |
| `sys_general_ledger` | — | `dv004` (turnover `CASE`/formula по layer) | — | a024/a025, GL-отчёты | ok (сложная метрика → dv по дереву выбора) |
| `p907_ym_payment_report` | — | `dv00x` | — | BI | проверить полноту |
| `p914_mp_finance_turnovers` | — | — | `d400` (вторично) | d400 | завязано на legacy d400 |

Заметки:
- `d4XX` — **отдельные дашборды** (потребители слоя), а НЕ обёртки над `ds01`/`ds02`. Неверной была
  пометка в CLAUDE.md «ds01/ds02 доступны как d401/d402» — исправлена. Коллизия двух `d401`
  устранена: `d401`=WB Finance (legacy, с бэкендом), метаданные перенесены на **`d405`**
  (`d402`/`d403`=история заказов WB/YM, `d404`=отчёт по рекламе WB). Внутренний CSS-namespace
  `.d401-*` метадашборда оставлен как косметический долг (общий `static/dashboards/d401.css`).
- Источник `ds03` и `dv001` над `p904` — НЕ конфликт, а разные роли; зафиксировать в обоих описаниях
  взаимную ссылку «гибкий аналог / курируемый аналог».

## Governance (схемы и сырой SQL равноправны, оба укреплены)

- **`dsNN`** — безопасны по построению: `QueryBuilder` ссылается только на поля схемы (таблицы/колонки
  — allowlist), параметры биндятся (`QueryParam`), `can_filter` проверяется в `condition_to_sql`.
  Можно отдавать LLM/плагинам как есть. Не выставлять авто-схемы секретных/служебных таблиц.
- **`dvNN`** — безопасны: метрики/измерения курируются в коде. Можно отдавать LLM/плагинам.
- **Raw SQL** — укреплённый escape-hatch. Чек-лист (для плана модернизации, переиспользовать
  существующее, не писать заново):
  1. Заменить подстрочный блок-лист в `execute_query`
     ([tool_executor.rs:911](crates/backend/src/shared/llm/tool_executor.rs#L911)) на
     токенную проверку `is_read_only_sql`
     ([bundle.rs:307](crates/contracts/src/plugins/bundle.rs#L307)) — она режет комментарии, 2-й
     statement и запрещённые ключевые слова токенами, а не подстроками.
  2. Применить табличный allowlist: `enforce_sql_capabilities` + `capability_allows_table`
     ([engine.rs:172](crates/backend/src/plugins/engine.rs#L172)) — явно **исключить** `sys_*` и
     connection-таблицы с секретами (`a001_connection_1c`, `a006_connection_mp`).
  3. Обернуть в `LIMIT` подзапросом (`limit_read_sql`, [engine.rs:200](crates/backend/src/plugins/engine.rs#L200))
     вместо текущей подстрочной правки LIMIT.
  4. Добавить таймаут запроса (сейчас нет).
- Итог: единый read-only валидатор и единая capability-модель для **обоих** SQL-путей (плагины и
  `execute_query`), вместо двух разных реализаций.

## Конвенции именования

- `dsNN` — базовые схемы (`ds01`…), таблица объявляется через `DSnn_TABLE_NAME`, схема — `DSnn_SCHEMA`.
- `dvNN` — курируемые виртуальные таблицы (`dv001`…), Rust-модуль `data_view/dvNNN/`.
- Каталог: `source_ref = { kind, id }`, где `id` = `dsNN` | `dvNN` | (для raw — нет id).
- Каждый источник несёт `description` (для людей и LLM) и взаимные ссылки при by-design перекрытии.

### UI-термины (2026-06-27)

Код-идентификаторы (`dsNN`/`dvNN`, `kind:"base"/"dataview"`, `data_schemes`/`data_view`) НЕ меняются;
в интерфейсе у них свои подписи, чтобы не путать два механизма:

- `dsNN` (base) → **«Схемы таблиц»** (каталог, `schema_browser`) + **«Конструктор запросов»**
  (построитель, `universal_dashboard`). Раньше: «Схемы данных (dsNN)» и «Конструктор по схемам».
- `dvNN` (dataview) → **«DataView»** (без изменений).
- Сайдбар-группа `semantic_layer` → **«Источники данных»** (раньше «Семантический слой»); перенесена
  в нижний блок сайдбара.

## Последствия

- (+) Прозрачные роли и единая витрина — потребитель/LLM выбирает источник по дереву.
- (+) Без миграции движков и без реестра метрик — минимум работы и риска.
- (+) Governance унифицируется на существующих примитивах; закрывается чтение секретов сырым SQL.
- (−) Сохраняется ограниченное дублирование определений (осознанное, задокументированное).
- (−) Нужна дисциплина «правила перекрытия», иначе дубли расползутся — контролируется через каталог
  и обязательные взаимные ссылки.

## Что вне Фазы 0 (план модернизации)

Реализация каталога/фасада; LLM-инструмент чтения строк (`query_schema`/`run_data_view`); plugin
host `host.data.query`; источник дата-блоков презентаций; укрепление `execute_query` по чек-листу
выше; доведение `QueryBuilder` (LIMIT, починка `sort.rules`
[query_builder.rs:470](crates/backend/src/shared/universal_dashboard/query_builder.rs#L470)); ревизия
хардкод-`d400` (дубль агрегаций над `p903`/`p914`); раскатка авто-схем (`metadata_to_pivot_schema`)
на разрешённые таблицы.
