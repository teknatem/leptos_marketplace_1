# A026 WB Advert Daily — Учёт рекламных расходов WB

`a026_wb_advert_daily` хранит суточные данные по рекламным кампаниям Wildberries:
расход по каждой кампании, метрики (просмотры, клики, заказы) и атрибуцию расходов к заказам.

## Проблема: временной разрыв

WB списывает деньги за клики в день X, но реальные продажи по этим кликам
могут состояться через 2–5 дней. Признавать рекламный расход в день X искажает
маржу: в день X маржа занижена, в дни X+2..X+5 — завышена.

## Решение: двухфазная GL-схема через счёт 9601

Вместо немедленного отнесения на расход используется счёт-резервуар 9601:

```
Phase 1 (постинг a026):   Д9601 / К7609 — резерв рекламного расхода
Phase 2 (постинг a012):   Д9102 / К9601 — списание резерва при реализации
```

- **7609** — расчёты с маркетплейсом (кредиторская задолженность WB).
- **9601** — транзитный счёт «Резерв рекламных расходов».
- **9102** — прямые переменные затраты на продажу (рекламные расходы при реализации).

## Проекция p913: order-line уровень атрибуции

`p913_wb_advert_order_attr` — вспомогательная таблица, которая связывает
конкретные заказы (srid из a012) с рекламными расходами (из a026).

Поля:
| Поле | Описание |
|------|---------|
| `turnover_code` | `advert_clicks_order_accrual` (Phase 1) или `advert_clicks_order_expense` (Phase 2) |
| `order_key` | srid заказа из a012; пусто при `is_problem=true` |
| `wb_advert_campaign_code` | Код кампании WB (advert_id) |
| `amount` | Доля расхода, атрибутированная этому заказу |
| `registrator_type` | `a026_wb_advert_daily` (Phase 1) или `a012_wb_sales` (Phase 2) |
| `registrator_ref` | UUID документа-регистратора |
| `is_problem` | `true` — расход есть, но заказ не найден |

## Атрибуция заказов к кампании

При постинге a026 система ищет заказы a015 за ту же дату по nm_id позиций кампании.
Суммарный рекламный расход распределяется пропорционально `finished_price` заказа.

- Выбирается первые N заказов (N = `wb_reported_orders` по данным WB).
- Остаток сверх N виден в таблице, но получает `is_allocated=false`.
- Если для nm_id нет заказов — создаётся строка `is_problem=true`.

## Схема данных

```
a026_wb_advert_daily (документ)
  ↓ posting Phase 1
  p913_wb_advert_order_attr (advert_clicks_order_accrual строки per-order)
  GL: Д9601 / К7609

a012_wb_sales (документ реализации)
  ↓ posting Phase 2
  p913_wb_advert_order_attr (advert_clicks_order_expense строки)
  GL: Д9102 / К9601
```

## Известные ограничения

- **Порядок импорта**: если a012 проведён раньше a026, reserve-строки ещё не существуют
  и expense не создаётся. При перепроведении a012 после a026 — всё сходится автоматически.
- **Возвраты** (`is_customer_return=true` в a012): expense не создаётся, reserve остаётся.
  Анализируется при закрытии периода как variance.
- **Отмена проведения a026**: удаляются только reserve-строки этого документа.
  Expense-строки связанных a012 не удаляются автоматически.

## Связанные документы

- `a012_wb_sales` — строки продаж WB; Phase 2 постинга.
- `p913_wb_advert_order_attr` — проекция с per-order атрибуцией.
- `general_ledger` — журнал GL-проводок по оборотам `advert_clicks_order_accrual` и `advert_clicks_order_expense`.
