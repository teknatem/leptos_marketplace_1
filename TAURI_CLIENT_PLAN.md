# TAURI_CLIENT_PLAN.md — план первой версии десктоп-клиента

> Документ самодостаточен и предназначен для переноса в новый репозиторий `mpi-desktop`.
> Память Claude привязана к каталогу серверного репо и в новый репо НЕ переедет — этот файл её заменяет.
> Дата: 2026-06. Серверный репо: `leptos_marketplace_1` (Rust/Axum + Leptos CSR/WASM).

---

## 1. Цель

Десктоп-оболочка (Tauri) как **тонкий клиент** к существующему многопользовательскому серверному приложению. Сервер (Axum:3000, SQLite, JWT, плановые задачи, интеграции WB/YM/Ozon, внешний API для 1С) **остаётся как есть**. Оболочка добавляет нативные возможности, которых нет у браузера, и удобное переключение между несколькими бэкендами.

Приложение **не переводится в локальное/однопользовательское** — бэкенд должен жить на сервере (плановые cron-импорты хотят 24/7; SQLite не любит конкуренцию; 1С-хаб централизован).

---

## 2. Архитектурное решение

### Модель C — гибрид (выбрано)
- Tauri несёт только **тонкую локальную оболочку**: окно(а), выбор профиля подключения, нативное меню/трей, single-instance, состояние окна. Это доверенный локальный HTML, бандлится в приложение, меняется редко.
- **Основной вебвью грузит весь `dist` (html+js+wasm+css+static+assets) С СЕРВЕРА** (`https://server/`), как сегодня браузер. WASM/CSS/статика **НЕ зашиты** в клиент.

### Следствия
- **Обновление фронта мгновенное**: деплой `dist` на сервер — как сейчас. Клиент-оболочка обновляется отдельно и редко (Tauri-updater, фаза 2).
- **`api_base()` править НЕ нужно**: origin = сервер, `window.location.host` валиден. (Правка понадобится только при переходе на бандл-модель.)
- **Офлайн теряется** — для тонкого клиента к серверу это приемлемо.

### Отдельный репозиторий (выбрано): `mpi-desktop`
Для Модели C клиент не бандлит фронт и не зависит от `contracts`/кода фронта — связь только по HTTP (API + base_url). Нет связи на уровне сборки → отдельный репо: свой релизный цикл, Tauri/WebView2/node-тулинг вне серверного workspace.
*Монорепо вернулось бы только при переходе на Модель A (бандл) для офлайн/закрытого контура.*

## 3. Факты о серверной стороне (проверено в коде)

| Что | Состояние | Где |
|-----|-----------|-----|
| CORS | `allow_origin(Any)` — origin вебвью примет без правок | `crates/backend/src/main.rs:160` |
| Auth | JWT **Bearer в заголовке** (не cookie); fetch-interceptor шьёт `Authorization` во все `/api/` | `dist/index.html`, `crates/frontend/src/shared/auth_download.rs` |
| Раздача фронта | `fallback_service(ServeDir::new("dist"))` отдаёт весь фронт | `crates/backend/src/main.rs:177` |
| Статика | CSS/ассеты — относительные пути (`static/...`, `assets/...`), копируются Trunk'ом (`copy-dir` в `crates/frontend/index.html`) | — |
| Excel-парсер | **self-host выполнен**: `static/xlsx.mini.min.js` (270 КБ, read-only), CDN убран | `crates/frontend/index.html` |

**Вывод:** бэкенд к тонкому клиенту уже готов. Cookie-сессий нет → cross-origin без боли. CSP-блокеров от внешнего CDN больше нет.

### Остаётся сделать на серверной стороне (этот репо, отдельно от mpi-desktop)
- (Опц.) включить gzip-сжатие статики: фича `compression-gzip` у `tower-http` (сейчас только `cors`, `fs`).
- (Фаза 2) CSP-заголовки под webview-origin; admin-API для функций администрирования; JS-хуки для нативных host-функций (save/notify).

## 4. Как работает клиент (поток)

```
запуск app
  └─ Rust читает активный профиль из store
       ├─ есть профиль → главное окно грузит https://<profile.base_url>/  (фронт с сервера)
       └─ нет профиля  → открыть окно "Подключения" (локальное, доверенное)

окно "Подключения" (локальный бандл): список профилей {name, base_url, env-цвет}
  └─ выбор → сохранить активный → главное окно navigate на сервер; заголовок = name
        (повторно доступно из нативного меню и трея для смены бэка)
```

- **Два окна:** главное (грузит сервер) и «Подключения» (локальное). Разделение = чистая граница безопасности: удалённая страница не имеет доступа к окну управления.
- Смена бэка = navigate главного окна + ре-логин (JWT в `localStorage` origin сервера; токены разных бэков не смешиваются, т.к. это разные origin). Для MVP ре-логин при переключении — норм.
- Цветовая метка/заголовок окна по `env` профиля (prod/staging) — против путаницы боевой и тестовой.

## 5. Объём MVP (первая версия)

| # | Функция | Примечание |
|---|---------|-----------|
| 1 | Tauri-оболочка, главное окно грузит `https://server/` | WebView2 (Win) |
| 2 | **Менеджер профилей подключения** (add/edit/delete/select, persist) — флагманская ценность | локальное окно + `tauri-plugin-store` |
| 3 | Navigate главного окна на активный бэк + заголовок/цвет по профилю | — |
| 4 | Single-instance (от «несколько копий → БД locked») | `tauri-plugin-single-instance` |
| 5 | Сохранение размеров/положения окна | `tauri-plugin-window-state` |
| 6 | Базовое нативное меню: Подключения, Перезагрузить, DevTools, Выход | `tauri::menu` |
| 7 | Инсталлятор MSI/NSIS | tauri-bundler |

**Граница безопасности MVP:** удалённой странице **НЕ выдаём никакого Tauri-IPC** (в Tauri v2 capabilities скоупятся на окна — главному окну не давать ни одной capability). Нативные функции живут только в локальном окне «Подключения». → нулевой remote-IPC риск, фронт обновляется мгновенно.

## 7. Пошаговая реализация

1. **Каркас.** `npm create tauri-app@latest mpi-desktop` (vanilla TS/JS, без фреймворка — оболочка крошечная). Tauri **v2**.
2. **Плагины.** Добавить `tauri-plugin-store`, `tauri-plugin-single-instance`, `tauri-plugin-window-state`. Зарегистрировать в `main.rs`.
3. **Модель профиля.** `Profile { id, name, base_url, env: "prod"|"staging"|"other" }`. Хранить в store (`profiles.json`) + ключ `active_profile_id`.
4. **Команды** (`commands.rs`, `#[tauri::command]`): `list_profiles`, `save_profile`, `delete_profile`, `select_profile(id)`, `get_active`. `select_profile` → сохранить active → `main_window.navigate(base_url)` + `set_title(name)`.
5. **Окно «Подключения»** (`src/`): рендер списка, форма add/edit, кнопка «Подключиться». Вызовы команд через `@tauri-apps/api/core invoke`.
6. **Главное окно.** В `setup`: если есть active → `navigate(base_url)`; иначе показать окно «Подключения». Главному окну **не назначать capabilities**.
7. **Меню/трей.** Пункты: «Подключения» (показать локальное окно), «Перезагрузить», «DevTools» (dev), «Выход». (Трей можно отложить в фазу 1.)
8. **Single-instance + window-state.** Подключить плагины.
9. **Сборка.** `npm run tauri build` → MSI. Проверить наличие WebView2 runtime (на Win11 обычно есть; иначе bootstrapper в инсталлятор).

## 8. Безопасность

- **MVP:** удалённой странице — ноль Tauri-IPC (никаких capabilities на главном окне). Локальное окно «Подключения» — отдельная capability только на свои команды.
- **Фаза 1+ (нативные host-функции для серверной страницы — save/notify):** требует **явной выдачи scoped-IPC доверенным backend-origin'ам**. Это supply-chain-риск (JS сервера получает нативный доступ). Митигировать: только HTTPS, белый список известных origin'ов, минимальный набор разрешённых команд, запись файлов только в песочный exports-каталог.
- **CSP** (серверная сторона): при включении строгого CSP под вебвью убедиться, что `static/xlsx.mini.min.js`, темы и ассеты в `script-src`/`style-src` разрешены (теперь все локальные, внешних нет).

## 9. Что НЕ в MVP

**Фаза 1 (дёшево, высокая ценность):**
- Keychain (Windows Credential Manager) для JWT/refresh вместо `localStorage`.
- Нативный «Сохранить как…» для экспорта — централизованно через единственный хелпер `crates/frontend/src/shared/auth_download.rs` (не 20 мест). Требует scoped-IPC (см. §8).
- Нативные тосты о завершении импортов (u502–u507) и репоста (u508).
- Системный трей + автозапуск.

**Фаза 2:**
- Health/версия/latency по каждому бэку; рассинхрон версии фронта vs API.
- Авто-обновление оболочки (`tauri-plugin-updater`, нужен ключ подписи + манифест на сервере).
- Хуки администрирования через защищённый admin-API.
- Plugin dev-loop: file-watch папки плагина → автоперезагрузка (см. подсистему Plugin в серверном репо).

## 10. Риски и открытые вопросы

- **WebView2 runtime** на машинах операторов (Win11 — обычно есть; иначе bootstrapper).
- **Navigate на remote в Tauri v2**: подтвердить API навигации окна на внешний URL и что главное окно остаётся без IPC.
- **Смешение токенов** при мультибэкенде: разные origin → разные `localStorage`, не смешиваются. Подтвердить на тесте переключения.
- **HTTPS на сервере**: для scoped-IPC (фаза 1) и в целом — желателен reverse-proxy с TLS (в текущем деплое сервер на `http://...:3000`).
- **Вес**: бандл оболочки маленький (фронт с сервера); проверить итоговый размер MSI.

## 11. Чеклист приёмки MVP

- [ ] Приложение запускается; при отсутствии профиля открывается окно «Подключения».
- [ ] Можно добавить/изменить/удалить профиль; они сохраняются между запусками.
- [ ] Выбор профиля грузит фронт с нужного сервера; заголовок окна = имя профиля.
- [ ] Логин на боевой бэк работает; список → деталь → один импорт → один экспорт проходят.
- [ ] Переключение на другой бэк работает (через меню «Подключения»), без смешения сессий.
- [ ] Вторая копия приложения не запускается (single-instance).
- [ ] Размер/положение окна восстанавливаются.
- [ ] Собран MSI; установка на чистой Win11 проходит.
